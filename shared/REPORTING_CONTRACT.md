# Reporting Contract

## Purpose
This contract standardizes how progress, blockers, critique, and completion are reported.

## Reporting principles
- Reports are concise, specific, and evidence-based.
- Completion claims are tied to observable outputs.
- Blockers are surfaced early, with the next decision made explicit.
- Reports separate what was done from what remains uncertain.
- A role does not use "done" for draft-only, partial, or unverified states.

## Standard report shapes

### Kickoff report
- objective
- scoped plan
- known risks or missing inputs
- approval needs, if any

### Progress report
- current stage
- completed
- in progress
- last error, if any
- next action
- ETA or confidence note
- approval needed? yes or no

### Critique report
- review target
- blocking defects
- non-blocking improvements
- concrete fix tickets
- evidence checked
- recommendation: revise, hold, or ready-for-verify

### Completion report
- delivered outcome
- evidence
- files, artifacts, or systems touched
- unresolved items
- recommended next action

### Failure or stop report
- stop reason
- attempted remediation
- exact blocker
- smallest decision needed

## Evidence rule
Evidence may include changed files, source links, test results, command outcomes, screenshots, or clearly described observations. If evidence is unavailable, the report states that explicitly.

## Overclaiming rule
A role does not report "done" when the work is only drafted, partially applied, step-complete, or unverified.
